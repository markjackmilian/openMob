## Executive Summary

This feature introduces a "New Session" button in the lateral drawer, positioned above the existing session list. When tapped, the button creates a new empty session on the currently active project (auto-selecting the first available project if none is active) and navigates the user directly to the `ChatPage` ready to start chatting. The session title starts as a placeholder and is updated automatically after the first message via opencode's `title` subagent, with the update reflected in real time in both the drawer and the chat header.

---

## Scope

### In Scope
- "New Session" button in the lateral drawer, positioned above the session list
- Auto-selection of the first available project from the server when no project is currently active
- Session creation via opencode API (`POST /session` with active `projectId`)
- Automatic navigation to `ChatPage` with the newly created session loaded and ready for input
- Session created with an empty/placeholder title — no user input required
- Real-time title update after the first message: openMob listens to the SSE event carrying the title generated by opencode's `title` subagent and updates the session title in the drawer list and `ChatPage` header without a page reload
- Loading state on the button during creation (disabled + visual indicator) to prevent duplicate taps
- Inline error message in the drawer on creation failure
- Drawer session list updated immediately after creation (new session appears at the top)
- `NewSessionCommand` on `DrawerViewModel` (or equivalent drawer ViewModel)
- New `CreateSessionAsync(string projectId)` method on `ISessionService`

### Out of Scope
- Manual title input by the user
- Session creation when no server is connected/reachable
- Server-side `title` subagent logic (handled entirely by opencode)
- Any other changes to the drawer layout, styles, or navigation items beyond the new button
- Deletion or renaming of sessions from the drawer (separate feature)

## Functional Requirements

> Requirements are numbered for traceability.

1. **[REQ-001]** The lateral drawer must display a "New Session" button above the session list. The button is always visible regardless of whether a project is active.

2. **[REQ-002]** When the user taps "New Session" and a project is currently active, the session is created on that project without any additional user interaction.

3. **[REQ-003]** When the user taps "New Session" and no project is currently active, the app must:
   a. Fetch the list of available projects from the server.
   b. Set the first project in the returned list as the active project.
   c. Proceed with session creation on that project.
   If no projects are available, display an inline error message and abort (no navigation).

4. **[REQ-004]** Session creation is performed by calling the opencode API (`POST /session`) with the active `projectId`. The API response must return the new `sessionId`.

5. **[REQ-005]** The new session is created with an empty or placeholder title (e.g., `"New Session"` or empty string). No title input is requested from the user at creation time.

6. **[REQ-006]** Immediately after successful creation, the app closes the drawer and navigates to `ChatPage` with the new session loaded. The chat input field must be focused and ready for the user's first message.

7. **[REQ-007]** The new session must appear at the top of the session list in the drawer immediately after creation, before any title is assigned.

8. **[REQ-008]** While the creation operation is in progress (from tap to successful navigation), the "New Session" button must be disabled and display a loading indicator to prevent duplicate session creation.

9. **[REQ-009]** If the creation API call fails (network error, server error, no projects available), the button returns to its normal state and an inline error message is displayed within the drawer. No navigation occurs.

10. **[REQ-010]** After the user sends the first message in the new session, opencode's `title` subagent generates a title server-side and emits an SSE event. openMob must handle this SSE event and update the session title in:
    - The drawer session list item for that session.
    - The `ChatPage` header title.
    Both updates must happen without a full page reload or manual refresh.

11. **[REQ-011]** The `DrawerViewModel` (or equivalent) must expose:
    - `NewSessionCommand` (`[AsyncRelayCommand]`) — triggers the full creation flow.
    - `IsCreatingSession` (`bool`, `[ObservableProperty]`) — true while the API call is in progress; drives button disabled state and loading indicator.
    - `CreationError` (`string?`, `[ObservableProperty]`) — set on failure, cleared at the start of each new attempt.

12. **[REQ-012]** `ISessionService` must be extended with a new method:
    ```
    Task<SessionDto> CreateSessionAsync(string projectId, CancellationToken ct = default)
    ```
    The method calls `POST /session`, deserialises the response into a `SessionDto`, and returns it. On non-success HTTP status, it throws an exception with a user-readable message.

### Key Findings from Codebase Exploration

#### 1. DrawerViewModel = `FlyoutViewModel`
The lateral drawer is backed by `FlyoutViewModel` (`src/openMob.Core/ViewModels/FlyoutViewModel.cs`). It already has:
- `Sessions: ObservableCollection<SessionItem>` — the session list
- `NewChatCommand` — creates a session via `_sessionService.CreateSessionAsync(null, ct)` and navigates to `//chat`
- `IsLoading: bool` — loading state for session list loading
- Subscriptions to `SessionDeletedMessage`, `CurrentSessionChangedMessage`, `ActiveProjectChangedMessage`

The existing `NewChatCommand` does NOT pass a `projectId` — it calls `CreateSessionAsync(null, ct)`. The spec requires project-aware creation with auto-selection fallback and inline error display (not a popup).

#### 2. `ISessionService.CreateSessionAsync` — existing signature
Current: `Task<SessionDto?> CreateSessionAsync(string? title, CancellationToken ct = default)`
The spec requires a new overload: `Task<SessionDto> CreateSessionAsync(string projectId, CancellationToken ct = default)` that throws on failure (instead of returning null).

**Decision:** Add a new method `CreateSessionForProjectAsync(string projectId, CancellationToken ct = default)` to `ISessionService` that:
- Calls `POST /session` with `CreateSessionRequest(Title: null, ParentId: null)` — the server uses the active project context
- BUT the opencode API does NOT accept `projectId` in the request body (confirmed: `CreateSessionRequest` only has `Title` and `ParentId`)
- **Critical finding:** The opencode `POST /session` endpoint does NOT take a `projectId` in the body. The server creates the session on the currently active project server-side. Therefore, the app must first call `IActiveProjectService.SetActiveProjectAsync(projectId)` to set the active project on the client side, then call `CreateSessionAsync`.

**Revised approach for `NewSessionCommand`:**
1. Get active project via `_activeProjectService.GetActiveProjectAsync(ct)`
2. If null → fetch all projects via `_projectService.GetAllProjectsAsync(ct)`, take first, call `_activeProjectService.SetActiveProjectAsync(firstProject.Id, ct)`
3. If still no project → set `CreationError`, return
4. Call `_sessionService.CreateSessionAsync(null, ct)` (existing method — no projectId needed in API call)
5. Prepend new session to `Sessions` collection
6. Navigate to `//chat` with `sessionId`

This means **no new method on `ISessionService`** is needed for the API call itself. The spec's [REQ-012] asks for `CreateSessionAsync(string projectId)` but the API doesn't support projectId in the body. The correct implementation is to ensure the active project is set before calling the existing `CreateSessionAsync(null, ct)`.

**However**, the spec explicitly states [REQ-012]: `Task<SessionDto> CreateSessionAsync(string projectId, CancellationToken ct = default)` that throws on failure. We will add this method to `ISessionService` as a higher-level orchestration method that:
- Validates `projectId`
- Calls `_apiClient.CreateSessionAsync(new CreateSessionRequest(null, null), ct)`
- Throws `InvalidOperationException` with user-readable message on failure (instead of returning null)
- Returns `SessionDto` (non-nullable) on success

#### 3. SSE Title Update — already handled
`ChatViewModel.HandleSessionUpdated(SessionUpdatedEvent e)` already updates `SessionName` when a `session.updated` SSE event arrives. The `ChatPage` header is already bound to `SessionName`. **No changes needed to `ChatViewModel` or `ChatPage.xaml`.**

For the **drawer** to update the session title in the list, `FlyoutViewModel` needs to react to session title changes. The cleanest approach (consistent with established patterns) is:
- Introduce a new `SessionTitleUpdatedMessage(string SessionId, string NewTitle)` message
- Have `ChatViewModel.HandleSessionUpdated` publish this message via `WeakReferenceMessenger`
- Have `FlyoutViewModel` subscribe to `SessionTitleUpdatedMessage` and update the matching `SessionItem` in `Sessions`

#### 4. `CreateSessionRequest` — no changes needed
The existing `CreateSessionRequest(Title: null, ParentId: null)` is sufficient. The server creates the session on the currently active project. No `projectId` field exists in the opencode API request body.

#### 5. Navigation pattern — confirmed
`FlyoutViewModel.SelectSessionAsync` uses `_navigationService.GoToAsync("//chat", { "sessionId": sessionId }, ct)`. The new `NewSessionCommand` will use the same pattern.

#### 6. Existing `NewChatCommand` in `FlyoutHeaderView`
The existing `NewChatCommand` in `FlyoutHeaderView.xaml` (pencil icon in the header) will remain unchanged. The spec adds a **new** "New Session" button in `FlyoutContentView.xaml` above the session list, bound to a new `NewSessionCommand`. The existing `NewChatCommand` is a separate entry point that continues to work as before.

### Technical Risks

- **`NewSessionCommand` vs `NewChatCommand` coexistence:** Both commands exist on `FlyoutViewModel`. The new `NewSessionCommand` has project-aware logic and inline error display; the old `NewChatCommand` uses popup errors. They must not conflict. The existing `NewChatCommand` in `FlyoutHeaderView` is kept as-is.
- **Session list prepend:** After creation, the new session must appear at the top of `Sessions`. Since `Sessions` is an `ObservableCollection`, we must `Insert(0, newItem)` on the UI thread via `_dispatcher.Dispatch(...)`.
- **`SessionTitleUpdatedMessage` cross-test pollution:** Like `SessionDeletedMessage`, this new message must be handled carefully in tests. `FlyoutViewModelTests` is already in `MessengerTestCollection` — new tests must remain in the same collection.
- **`CreateSessionForProjectAsync` throws vs returns null:** The new method throws `InvalidOperationException` on failure. `FlyoutViewModel.NewSessionCommand` must catch this and set `CreationError` (not call `_popupService.ShowErrorAsync`).
